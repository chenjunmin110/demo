// 在浏览器输入URL浏览器内部执行了什么操作： 比如：https://www.baidu.com/ 1.解析URL url: 统一资源定位符，简称网址， https：传输协议，加密的http协议，就是在http和TCP之间加了一层TSL和SSL的安全层 www: 服务器 baidu.com: 域名 ？id=1&name='123': 参数 2.通过url解析出IP地址，DNS域名系统（DNS解析） url只是ip地址的一个映射，一一对应的，应为大家记一份ip地址都是点点，又是数字，脑壳疼 通常第一次访问完成之后，浏览器会将域名解析的映射缓存到本地 下次在访问该网点的时候，先从浏览器的缓存中看有没有这个域名解析过deIP地址，有没有过期等， 如果有这个ip地址而且没有过期，解析结束，如果还没有的话就会去操作系统的缓存中是否有对应域名的ip地址 3.浏览器与网站建立TCP（三次握手）（见pro图） 南方去女方家看妹子，客户端向服务器端发送请求（SYW=1），等待服务器确认; 女方家去男方家看家境，服务器收到请求确认，回复一个指令（SYW，ACK=1）; 大家都觉得合适那就找一家酒店订亲, 客户端收到服务器的回复指令并返回确认（ACK=1) 三次握手都是试探的，不携带任何数据（就是相亲） 4.请求和传输数据，渲染页面（见pro图） 这个后面谈----DOM树和CSS树并行解析，计算布局信息，再调用UI引擎 5.断开连接，第四次与TCP挥手，再见了你嘞 扩展一下 websocket // css加载会造成阻塞吗？ DOM树和CSS树并行解析，计算布局信息，再调用UI引擎, CSS加载不会阻塞DOM的解析 然而，由于Render Tree是依赖与DOM Tree 和CSS Tree的 所以必须等待到CSSDOM Tree构建完成 就是CSS资源加载完成后才能开始渲染 因此，CSS会阻塞DOM的渲染 由于Javascript是可以加载操作DOM和CSS的， 如果在修改这些元素属性同时渲染界面（即js和UI线程同时运行） 那么渲染的线程前后获取的元素数据可能不一致了，所以先执行GUI渲染再执行JS // 从那些点做性能优化 1.加载 a.减少http请求（精灵图，css和js，根据实际情况，进行内联与合并） b.减少文件大小（资源压缩，图片压缩，Gzip服务端压缩） c.CDN库（大图，大文件，js文件上传到CDN库链接引入） d.性能：减少DOM操作，避免回流，对于操作量很大的dom，采用文档碎片（虚拟dom原理） e.SRR服务端渲染，预渲染等等 f.路由懒加载，减少首屏加载量
